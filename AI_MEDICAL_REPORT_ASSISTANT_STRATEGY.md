### 1.2 مسار تدفق الرسالة الحالي (Message Flow)

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           مسار الرسالة الحالي                                      │
└─────────────────────────────────────────────────────────────────────────────────┘

[المستخدم] → [Frontend: ChatInput/ChatWindow]
                    │
                    ├─ نص ← handleSendMessage
                    ├─ صوت ← VoiceRecorder / SpeechRecognition
                    ├─ صورة ← onImageUpload → POST /api/extract-text (OCR)
                    └─ ملف ← onFileUpload → POST /api/extract-document
                    │
                    ▼
[API Router] → POST /api/chat  أو  POST /api/conversations/{id}/messages
                    │
                    ├─ Rate Limiter
                    ├─ Question Router (سعر → رد ثابت بدون API)
                    ├─ Smart Cache (إذا موجود → رد فوري)
                    │
                    ▼
[RAG Pipeline] → get_grounded_context()
                    │
                    ├─ Semantic Search (embeddings + cosine)
                    ├─ Lexical Search (rapidfuzz على الاسم، الوصف، الأعراض، التصنيف)
                    ├─ Merge + Threshold (0.58)
                    └─ has_relevant? → نعم: context | لا: NO_INFO_MESSAGE
                    │
                    ▼
[OpenAI Service] → generate_response()
                    │
                    ├─ System Prompt (قواعد صارمة، لا تشخيص، لا علاج)
                    ├─ Knowledge Context (من RAG فقط)
                    ├─ Conversation History (من DB)
                    └─ User Message
                    │
                    ▼
[Response] → حفظ في DB → إرجاع للمستخدم
```

### 1.3 ما هو موجود vs ما هو ناقص

| المتطلب | الموجود | الناقص |
|---------|---------|--------|
| **1. معلومات المختبر والفروع** | رقم تواصل 800-122-1220 في question_router | لا يوجد جدول للمختبر أو الفروع، أوقات العمل، خطوط الطوارئ |
| **2. دور المساعد** | System Prompt يحدد: لا تشخيص، لا علاج، معلومات فقط | لا يوجد تفسير للقيم، النطاقات المرجعية، أو بنية موحدة للردود |
| **3. أنواع الفحوصات** | 574 فحص في RAG (analysis_name_ar/en, description, symptoms, preparation, category, price) | لا تصنيف حسب التخصص (Clinical Chemistry, Hematology, etc.)، لا ربط تفصيلي بالنطاقات المرجعية |
| **4. القيود الحرجة** | لا تشخيص، لا علاج في الـ prompt | لا كشف كلمات الطوارئ، لا تصعيد تلقائي، لا Content Moderation |
| **5. إرشادات الردود** | أسلوب احترافي في الـ prompt | لا بنية موحدة (1–7)، لا مؤشرات بصرية 🟢🟡🔴، لا دعم ثنائي اللغة تلقائي |
| **6. إخلاء المسؤولية** | غير موجود | إخلاء مسؤولية إلزامي في كل رد |
| **7. أمثلة تدريبية** | RAG فقط (بدون Few-Shot) | لا Few-Shot في الـ prompt، لا بنك أمثلة |
| **8. النطاقات المرجعية والقيم الحرجة** | غير موجود | لا Rule Engine، لا كشف القيم الحرجة، لا تنبيهات |
| **9. تحليل التقارير متعددة الاختبارات** | OCR للروشتات (استخراج أسماء التحاليل) | لا تحليل PDF للتقارير الطبية، لا مقارنة، لا اتجاهات |
| **10. اعتبارات خاصة** | غير موجود | لا نطاقات للأطفال/الحوامل، لا حساسية ثقافية |
| **11. ضمان الجودة** | لا يوجد | لا Audit Trail، لا مراجعة طبية، لا امتثال |
| **12. واجهة المستخدم** | رفع صورة/ملف، عرض تفاعلي | لا عرض تفاعلي للنتائج مع ألوان، لا بوابة مرضى، لا إشعارات SMS/Email |

## 1️⃣ معلومات المختبر والفروع

### التصميم المقترح لقاعدة البيانات

```sql
-- جدول المختبر الرئيسي
CREATE TABLE laboratories (
    id UUID PRIMARY KEY,
    name_ar VARCHAR(255),
    name_en VARCHAR(255),
    address TEXT,
    specialty TEXT,
    accreditations JSONB,  -- ['ISO 15189', 'CAP']
    created_at TIMESTAMPTZ,
    updated_at TIMESTAMPTZ
);

-- جدول الفروع
CREATE TABLE branches (
    id UUID PRIMARY KEY,
    laboratory_id UUID REFERENCES laboratories(id),
    name_ar VARCHAR(255),
    address TEXT,
    phone VARCHAR(50),
    work_hours JSONB,  -- {"mon": "08:00-22:00", "tue": "08:00-22:00", ...}
    services JSONB,   -- ["CBC", "Liver Function", ...]
    emergency_line VARCHAR(50),
    support_after_hours VARCHAR(50),
    created_at TIMESTAMPTZ,
    updated_at TIMESTAMPTZ
);
```

### آلية الاستخدام في الشات بوت

- **إضافة طبقة جديدة** في RAG أو في `knowledge_loader`: تحميل معلومات المختبر والفروع من DB.
- **دمجها في System Prompt** أو في Context عند استفسار عن "الفرع الأقرب"، "أوقات العمل"، "خط الطوارئ".
- **دالة مساعدة** `get_lab_info_for_context()` تُستدعى عند كشف كلمات مثل "فرع"، "عنوان"، "أوقات"، "طوارئ".

## 2️⃣ دور المساعد الطبي وقدراته

### Prompt Engineering vs Fine-tuning

| الأسلوب | المزايا | العيوب | التوصية |
|---------|---------|--------|----------|
| **System Prompts** | سريع، قابل للتعديل، لا تكلفة إضافية | محدود بالتعليمات النصية | ✅ **استخدامه** للدور الأساسي |
| **Few-Shot** | يحسن السلوك دون تدريب | يزيد الـ tokens | ✅ **استخدامه** للردود النموذجية |
| **Fine-tuning** | سلوك مخصص جداً | تكلفة، صيانة، إعادة تدريب | ⚠️ لاحقاً عند النضج |

### بنية System Prompt المقترحة

```
أنت "وريد"، مساعد تفسير النتائج الطبية لمختبرات وريد.

=== دورك (فقط) ===
- شرح نتائج الفحوصات الفردية
- تفسير القيم مقارنة بالنطاقات المرجعية
- تحديد القيم غير الطبيعية
- شرح المصطلحات الطبية ببساطة
- تقديم معلومات عن التحضير للفحوصات

=== ما لا تفعله ===
- التشخيص الطبي
- التوصية بعلاجات أو أدوية
- اقتراح تغييرات في الأدوية
- نصائح طوارئ طبية
- قرارات سريرية

=== بنية الرد ===
1. الإقرار باستلام التقرير/السؤال
2. تحليل منهجي لكل معامل
3. شرح القيم في السياق
4. تسليط الضوء على القيم غير الطبيعية
5. تثقيف المريض
6. تجنب المصطلحات المعقدة
7. الختام بتوجيه لاستشارة الطبيب
```

## 3️⃣ أنواع الفحوصات والتخصصات

### Knowledge Base: Vector DB vs Structured DB

| الخيار | الاستخدام | التوصية |
|--------|-----------|----------|
| **Vector DB** (Pinecone, Weaviate, pgvector) | بحث دلالي، استرجاع مرن | ✅ للبحث الدلالي (موجود حالياً كـ embeddings JSON) |
| **Structured DB** (PostgreSQL) | فحوصات، نطاقات، تصنيفات | ✅ للبيانات المنظمة |

### التصميم المقترح

- **الإبقاء على RAG الحالي** للبحث الدلالي.
- **إضافة جداول منظمة** للتصنيفات والنطاقات:

```sql
CREATE TABLE test_categories (
    id UUID PRIMARY KEY,
    name_ar VARCHAR(255),
    name_en VARCHAR(255),
    parent_id UUID REFERENCES test_categories(id)
);

CREATE TABLE reference_ranges (
    id UUID PRIMARY KEY,
    test_id UUID,
    age_min INT, age_max INT,
    sex VARCHAR(20),  -- 'M', 'F', 'both'
    pregnancy BOOLEAN,
    unit VARCHAR(50),
    range_low DECIMAL,
    range_high DECIMAL,
    critical_low DECIMAL,
    critical_high DECIMAL
);
```

- **دمج التصنيفات** في `rag_knowledge_base.json` (حقل `specialty` أو `category_detail`).

## 4️⃣ القيود الحرجة وإرشادات السلامة

### Content Moderation و Safety Guardrails

1. **طبقة Pre-Processing** قبل استدعاء OpenAI:
   - كشف كلمات الطوارئ: `ألم صدري، صعوبة تنفس، نزيف شديد، إغماء، سكتة، جلطة، ...`
   - قائمة `EMERGENCY_KEYWORDS` في ملف config.

2. **آلية التصعيد**:
   ```
   إذا وُجدت كلمة طوارئ:
     → رد فوري ثابت: "هذه حالة طارئة. يرجى الاتصال بالإسعاف 911 أو التوجه لأقرب طوارئ فوراً."
     → لا استدعاء OpenAI
     → تسجيل في Audit Log
   ```

3. **Post-Processing** (اختياري):
   - فحص الرد النهائي للتأكد من عدم احتوائه على تشخيص أو علاج.
   - استخدام OpenAI Moderation API للكشف عن محتوى غير آمن.

### مثال كود (مفهومي)

```python
EMERGENCY_KEYWORDS = [
    "ألم صدري", "صعوبة تنفس", "نزيف شديد", "إغماء",
    "chest pain", "difficulty breathing", "severe bleeding"
]

def check_emergency(message: str) -> bool:
    normalized = message.lower().strip()
    return any(kw in normalized for kw in EMERGENCY_KEYWORDS)
```

## 5️⃣ إرشادات الردود والتواصل

### Response Templates

- **قالب موحد** يُحقن في System Prompt:
  - الإقرار → التحليل → الشرح → التنبيه → التثقيف → التوجيه للطبيب.

### المؤشرات البصرية

- إضافة تعليمات في الـ Prompt:
  - `🟢` للقيم الطبيعية
  - `🟡` للقيم قريبة من الحدود
  - `🔴` للقيم خارج النطاق

- **Frontend**: تحويل هذه الرموز إلى ألوان فعلية (مثلاً `<span class="normal">`، `<span class="warning">`، `<span class="critical">`).

### دعم ثنائي اللغة

- **اكتشاف اللغة** من أول رسالة (مكتبة `langdetect` أو من `Accept-Language`).
- **تبديل تلقائي** في الـ Prompt: "رد بالعربية" أو "Respond in English".
- تخزين تفضيل اللغة في `User` أو في الـ session.

## 6️⃣ إخلاء المسؤولية الطبية

### آلية التنفيذ

- **إضافة سطر ثابت** في نهاية كل رد من الـ LLM:
  - `"⚠️ هذا المساعد لا يقدم تشخيصاً أو علاجاً. استشر طبيبك دائماً."`

- **تنفيذ تقني**:
  - إما في Post-Processing بعد `generate_response()` بإلحاق النص.
  - أو في System Prompt: "اختتم كل رد بإخلاء المسؤولية التالي: ..."

- **إخلاء خاص بالسياق**:
  - عند وجود قيم غير طبيعية: إضافة جملة إضافية.
  - عند القيم الحرجة: إضافة تنبيه أقوى.

## 7️⃣ أمثلة تدريبية ومعرفة طبية

### Few-Shot Learning

- إضافة 3–5 أمثلة في System Prompt:

```
مثال 1:
المستخدم: لدي CBC طبيعي، ماذا يعني؟
المساعد: [رد نموذجي يشرح CBC ويؤكد أن النتائج طبيعية]

مثال 2:
المستخدم: إنزيمات الكبد مرتفعة لدي
المساعد: [رد نموذجي يشرح دون تشخيص]
```

### RAG المحسّن

- **الإبقاء على RAG الحالي** وتحسينه:
  - إضافة حقول `interpretation_guide`، `common_conditions` في قاعدة الفحوصات.
  - إثراء الـ embeddings بهذه الحقول.

## 8️⃣ النطاقات المرجعية والقيم الحرجة

### Rule Engine

- **جدول `critical_values`**:
  - test_code, unit, critical_low, critical_high, action (مثل "اتصل بالطبيب فوراً").

- **دالة `check_critical_values(results: List[Dict])`**:
  - تقارن كل قيمة بالنطاقات الحرجة.
  - إذا وُجدت قيمة حرجة → تنبيه فوري + إخلاء مسؤولية معزز.

- **بروتوكول التصعيد**:
  - إرسال تنبيه داخلي (Slack/Email) للطاقم الطبي.
  - إرجاع رد للمريض مع توجيه واضح للطوارئ أو الطبيب.

## 9️⃣ تحليل التقارير متعددة الاختبارات

### استخراج البيانات من PDF

- **الوضع الحالي**: `pypdf` يستخرج نصاً فقط (لا جداول منظمة).
- **الحلول**:
  - **Option A**: استخدام `pdfplumber` أو `camelot` لاستخراج الجداول.
  - **Option B**: استخدام Vision API (gpt-4o) لتحليل PDF كصورة واستخراج structured data.
  - **Option C**: تكامل مع أنظمة المختبر (HL7/FHIR) للحصول على بيانات منظمة.

### تحليل الاتجاهات

- تخزين التقارير في DB مع `patient_id` (مجهول)، `test_code`، `value`، `unit`، `date`.
- دالة `get_trends(patient_id, test_code)` تُرجع سلسلة زمنية.
- إضافة سياق الاتجاهات في الـ Prompt عند طلب "قارن مع التقرير السابق".

# الجزء السادس: التوصيات النهائية

1. **البدء بالمرحلة 1** دون تغيير البنية الحالية قدر الإمكان.
2. **تعزيز System Prompt** أولاً (أسرع وأقل تكلفة من Fine-tuning).
3. **إضافة طبقة Safety** (كشف الطوارئ) في أقرب وقت.
4. **بناء Rule Engine للقيم الحرجة** بالتوازي مع تحسين الـ Prompt.
5. **توثيق كل قرار** للامتثال والمراجعة المستقبلية.
